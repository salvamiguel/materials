# Workspaces y patrones avanzados

En esta unidad vamos a explorar las capacidades avanzadas de Terraform que nos permiten escalar nuestras configuraciones de infraestructura. Desde la gestión de múltiples entornos con workspaces hasta la integración en pipelines de CI/CD, estos conceptos son fundamentales para trabajar con Terraform en entornos profesionales.

---

## 1. Workspaces

### Qué son los workspaces

Un workspace en Terraform es una **instancia aislada de state** dentro del mismo directorio de configuración. Cuando ejecutamos `terraform init` por primera vez, Terraform crea automáticamente un workspace llamado `default`. Cada workspace mantiene su propio fichero de estado, lo que significa que los recursos gestionados en un workspace son completamente independientes de los gestionados en otro.

El caso de uso principal de los workspaces es **gestionar múltiples entornos** (dev, staging, prod) utilizando exactamente la misma configuración de Terraform, pero con diferentes valores y un state separado para cada uno.

### Comandos de gestión

Los comandos básicos para trabajar con workspaces son los siguientes:

```bash
# Listar todos los workspaces disponibles
terraform workspace list

# Crear un nuevo workspace (y cambiar a el automaticamente)
terraform workspace new dev

# Cambiar al workspace de produccion
terraform workspace select prod

# Mostrar el workspace activo
terraform workspace show

# Eliminar un workspace (debe estar vacio de recursos)
terraform workspace delete staging
```

Cuando creamos un nuevo workspace, Terraform genera un subdirectorio dentro de `terraform.tfstate.d/` con el nombre del workspace. El workspace `default` utiliza el fichero `terraform.tfstate` en la raíz del directorio.

### La variable terraform.workspace

Terraform expone una variable especial llamada `terraform.workspace` que contiene el nombre del workspace actualmente seleccionado. Esta variable es extremadamente útil para parametrizar nuestras configuraciones según el entorno.

### Workspaces vs directorios separados

:::info[Workspaces vs directorios separados]
Ambas estrategias son válidas para gestionar múltiples entornos, pero tienen trade-offs diferentes:

**Workspaces** son ideales cuando:
- La configuración es idéntica entre entornos (solo cambian variables).
- El equipo es pequeño y la coordinación es sencilla.
- Se quiere evitar duplicación de código.

**Directorios separados** son preferibles cuando:
- Los entornos tienen diferencias significativas en arquitectura.
- Se necesita control granular de permisos por entorno.
- Se quiere poder hacer `plan` o `apply` en un entorno sin riesgo de afectar otro.
- El equipo es grande y múltiples personas trabajan en paralelo.

En la práctica, muchos equipos profesionales prefieren directorios separados con módulos compartidos, ya que ofrece mayor aislamiento y reduce el riesgo de errores.
:::

## 2. for_each

### Iteración sobre maps y sets

El meta-argumento `for_each` permite crear múltiples instancias de un recurso iterando sobre un `map` o un `set` de strings. A diferencia de `count`, cada instancia se identifica por su clave, no por un índice numérico, lo que proporciona estabilidad ante cambios en la colección.

Dentro del bloque del recurso, tenemos acceso a dos valores especiales:

- `each.key` -- la clave del elemento actual del map (o el valor si es un set).
- `each.value` -- el valor del elemento actual del map.

### Ejemplo con map

<CloudTabs>
<CloudTab provider="aws">

```hcl
variable "instances" {
  description = "Mapa de instancias a crear con su configuracion"
  type = map(object({
    instance_type = string
    ami           = string
  }))
  default = {
    web = {
      instance_type = "t3.micro"
      ami           = "ami-0c55b159cbfafe1f0"
    }
    api = {
      instance_type = "t3.small"
      ami           = "ami-0c55b159cbfafe1f0"
    }
    worker = {
      instance_type = "t3.medium"
      ami           = "ami-0c55b159cbfafe1f0"
    }
  }
}

resource "aws_instance" "servers" {
  for_each      = var.instances
  ami           = each.value.ami
  instance_type = each.value.instance_type

  tags = {
    Name = each.key
    Role = each.key
  }
}

# Acceder a una instancia especifica por su clave
output "web_server_ip" {
  value = aws_instance.servers["web"].public_ip
}

# Acceder a todas las IPs
output "all_ips" {
  value = { for name, instance in aws_instance.servers : name => instance.public_ip }
}
```

### Ejemplo con set de strings

```hcl
variable "subnet_ids" {
  type = set(string)
  default = ["subnet-abc123", "subnet-def456", "subnet-ghi789"]
}

resource "aws_instance" "distributed" {
  for_each  = var.subnet_ids
  subnet_id = each.value
  # Cuando se itera sobre un set, each.key == each.value
}
```

### Ejemplo con toset y una lista

Cuando tenemos una lista y queremos usar `for_each`, debemos convertirla a set con `toset()`:

```hcl
variable "users" {
  type    = list(string)
  default = ["alice", "bob", "charlie"]
}

resource "aws_iam_user" "team" {
  for_each = toset(var.users)
  name     = each.value
}
```

### Diferencia entre for_each y count

| Característica | `count` | `for_each` |
|---|---|---|
| Identificador | Índice numérico (0, 1, 2...) | Clave del map o valor del set |
| Reordenamiento | Puede causar destroy/create | Estable ante cambios |
| Referencia | `resource.name[0]` | `resource.name["key"]` |
| Uso ideal | Recursos idénticos, condicionales | Recursos con identidad propia |

:::tip[Preferir for_each sobre count]
Siempre que los elementos tengan **identidad propia** (un nombre, un ID, un rol), usa `for_each` en lugar de `count`. Con `count`, si eliminamos el segundo elemento de una lista de cinco, los elementos 3, 4 y 5 cambian de índice, lo que provoca que Terraform destruya y recree esos recursos innecesariamente. Con `for_each`, solo se elimina el recurso correspondiente a la clave eliminada.
:::

## 3. count

### Crear N instancias de un recurso

El meta-argumento `count` es la forma más sencilla de crear múltiples copias de un recurso. Terraform asigna un índice numérico (empezando en 0) a cada instancia.

```hcl
variable "instance_count" {
  type    = number
  default = 3
}

resource "aws_instance" "cluster" {
  count         = var.instance_count
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name = "cluster-node-${count.index}"
  }
}

# Acceder a una instancia especifica
output "first_instance_ip" {
  value = aws_instance.cluster[0].public_ip
}

# Acceder a todas las IPs
output "all_instance_ips" {
  value = aws_instance.cluster[*].public_ip
}
```

### Patrón condicional con count

Uno de los usos más frecuentes de `count` es la **creación condicional** de recursos. Si `count = 0`, el recurso no se crea. Si `count = 1`, se crea una única instancia:

```hcl
variable "create_monitoring" {
  description = "Si es true, se crea la infraestructura de monitoring"
  type        = bool
  default     = false
}

resource "aws_cloudwatch_dashboard" "main" {
  count          = var.create_monitoring ? 1 : 0
  dashboard_name = "main-dashboard"
  dashboard_body = file("dashboard.json")
}

resource "aws_sns_topic" "alerts" {
  count = var.create_monitoring ? 1 : 0
  name  = "infra-alerts"
}

# Al referenciar un recurso condicional, debemos usar [0]
output "dashboard_arn" {
  value = var.create_monitoring ? aws_cloudwatch_dashboard.main[0].dashboard_arn : null
}
```

### Limitaciones de count

Es importante entender las limitaciones de `count` para evitar problemas:

1. **Dependencia del índice**: los recursos se identifican por su posición numérica. Si eliminamos un elemento del medio de la lista, todos los índices posteriores cambian.

2. **Reordering causa destroy/create**: cambiar el orden de los elementos puede provocar que Terraform destruya y recree recursos innecesariamente, lo cual es particularmente peligroso con bases de datos o recursos con datos persistentes.

3. **No se puede usar con `for_each`**: un recurso puede tener `count` o `for_each`, pero no ambos.

```hcl
# PROBLEMA: si eliminamos "web" de la lista,
# "api" pasa de indice 1 a 0 y "worker" de 2 a 1
# Terraform destruira y recreara ambos
variable "server_names" {
  default = ["web", "api", "worker"]
}

resource "aws_instance" "servers" {
  count = length(var.server_names)
  tags = {
    Name = var.server_names[count.index]
  }
}
```

En este caso, sería mucho mejor usar `for_each` con `toset(var.server_names)`.

## 4. Dynamic blocks

### Generar bloques repetidos dentro de un recurso

Los dynamic blocks permiten generar bloques de configuración repetidos dentro de un recurso de forma programática. Son útiles cuando un recurso necesita múltiples bloques del mismo tipo (como reglas de un security group, tags, o configuraciones de disco).

### Ejemplo: security group con múltiples reglas de ingress

<CloudTabs>
<CloudTab provider="aws">

```hcl
variable "ingress_rules" {
  description = "Lista de reglas de ingress para el security group"
  type = list(object({
    port        = number
    protocol    = string
    cidr_blocks = list(string)
    description = string
  }))
  default = [
    {
      port        = 80
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTP"
    },
    {
      port        = 443
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTPS"
    },
    {
      port        = 22
      protocol    = "tcp"
      cidr_blocks = ["10.0.0.0/8"]
      description = "SSH desde red interna"
    }
  ]
}

resource "aws_security_group" "main" {
  name        = "main-sg"
  description = "Security group principal"
  vpc_id      = aws_vpc.main.id

  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = ingress.value.protocol
      cidr_blocks = ingress.value.cidr_blocks
      description = ingress.value.description
    }
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

### Anatomía de un dynamic block

La estructura de un dynamic block tiene tres partes:

```hcl
dynamic "NOMBRE_DEL_BLOQUE" {
  for_each = COLECCION_A_ITERAR
  content {
    # Acceder a cada elemento con NOMBRE_DEL_BLOQUE.value
    # Acceder a la clave con NOMBRE_DEL_BLOQUE.key
  }
}
```

Se puede cambiar el nombre del iterador con el argumento `iterator` si se necesita mayor claridad o para evitar conflictos:

```hcl
dynamic "ingress" {
  for_each = var.ingress_rules
  iterator = rule
  content {
    from_port   = rule.value.port
    to_port     = rule.value.port
    protocol    = rule.value.protocol
    cidr_blocks = rule.value.cidr_blocks
  }
}
```

### Dynamic blocks anidados

Es posible anidar dynamic blocks, aunque esto debe hacerse con cuidado:

```hcl
variable "load_balancer_config" {
  type = list(object({
    name     = string
    listeners = list(object({
      port     = number
      protocol = string
    }))
  }))
}

resource "aws_lb" "main" {
  for_each = { for lb in var.load_balancer_config : lb.name => lb }
  name     = each.key

  dynamic "subnet_mapping" {
    for_each = var.subnet_ids
    content {
      subnet_id = subnet_mapping.value
    }
  }
}
```

:::warning[No abusar de dynamic blocks]
Los dynamic blocks son una herramienta poderosa, pero su uso excesivo puede reducir significativamente la **legibilidad** de la configuración. Antes de usar un dynamic block, pregúntate si la repetición manual de los bloques sería más clara. Reserva los dynamic blocks para casos donde la lista de bloques es verdaderamente variable o extensa. Recuerda que el código de infraestructura debe ser fácil de auditar.
:::

## 5. Expresiones condicionales

### Operador ternario

Terraform soporta el operador ternario clásico para evaluar condiciones inline:

```hcl
# Sintaxis: condicion ? valor_si_true : valor_si_false

instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"
```

### Ejemplos prácticos

Las expresiones condicionales son especialmente útiles para ajustar la configuración según el entorno o las necesidades:

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

variable "enable_monitoring" {
  type    = bool
  default = true
}

resource "aws_instance" "app" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"

  # Activar monitoring detallado solo en produccion
  monitoring = var.environment == "prod" ? true : false

  # Asignar IP publica solo en dev
  associate_public_ip_address = var.environment == "dev" ? true : false

  # Elegir subnet segun el entorno
  subnet_id = var.environment == "prod" ? var.private_subnet_id : var.public_subnet_id

  root_block_device {
    volume_size = var.environment == "prod" ? 100 : 20
    volume_type = var.environment == "prod" ? "gp3" : "gp2"
    encrypted   = var.environment == "prod" ? true : false
  }
}
```

### Combinar condicionales con count

El patrón más común es usar el operador ternario con `count` para crear o no un recurso:

```hcl
variable "environment" {
  type = string
}

# Solo crear el bastion host en entornos que no sean produccion

  count         = var.environment != "prod" ? 1 : 0
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

    Name = "bastion-${var.environment}"
  }
}

  count   = var.environment == "prod" ? 1 : 0
  enabled = true
  # ... configuracion completa
}
```

## 6. Funciones built-in

### Funciones en Terraform

Terraform incluye un amplio conjunto de **funciones built-in** que permiten transformar y combinar valores dentro de las configuraciones. Es importante saber que Terraform **no soporta funciones custom** definidas por el usuario; solo se pueden usar las funciones que vienen integradas en el lenguaje.

### Funciones de String

```hcl
# Convertir a mayusculas/minusculas
upper("hello")          # "HELLO"
lower("Hello World")    # "hello world"

# Formatear cadenas (similar a printf)
format("Hello, %s!", "world")           # "Hello, world!"
format("instance-%03d", 5)              # "instance-005"

# Unir y separar
join(", ", ["a", "b", "c"])             # "a, b, c"
split(",", "a,b,c")                     # ["a", "b", "c"]

# Reemplazar
replace("hello world", "world", "terraform")  # "hello terraform"

# Eliminar espacios en blanco
trimspace("  hello  ")                  # "hello"

# Subcadenas
substr("hello world", 0, 5)            # "hello"
```

### Funciones de Collection

```hcl
# Longitud
length(["a", "b", "c"])                # 3
length({a = 1, b = 2})                 # 2

# Buscar en un map con valor por defecto
lookup({env = "prod", region = "eu"}, "env", "dev")        # "prod"
lookup({env = "prod", region = "eu"}, "missing", "default") # "default"

# Combinar maps (los ultimos sobreescriben)
merge({a = 1, b = 2}, {b = 3, c = 4})  # {a = 1, b = 3, c = 4}

# Aplanar listas anidadas
flatten([["a", "b"], ["c"], ["d", "e"]])  # ["a", "b", "c", "d", "e"]

# Obtener claves y valores de un map
keys({a = 1, b = 2, c = 3})            # ["a", "b", "c"]
values({a = 1, b = 2, c = 3})          # [1, 2, 3]

# Crear un map a partir de dos listas
zipmap(["name", "age"], ["Alice", 30])  # {name = "Alice", age = 30}

# Concatenar listas
concat(["a", "b"], ["c", "d"])          # ["a", "b", "c", "d"]

# Filtrar elementos (con expresion for)
[for s in ["a", "b", "c", "d"] : s if s != "c"]  # ["a", "b", "d"]
```

### Funciones numéricas

```hcl
min(5, 3, 8, 1)     # 1
max(5, 3, 8, 1)     # 8
ceil(4.3)            # 5
floor(4.7)           # 4
abs(-15)             # 15
```

### Funciones de Filesystem

```hcl
# Leer el contenido de un fichero
file("${path.module}/config.json")

# Renderizar un template con variables

```

## 7. Provisioners

### Qué son los provisioners

Los provisioners son mecanismos que permiten **ejecutar scripts o comandos** en recursos después de su creación (o antes de su destrucción). Terraform ofrece dos tipos principales:

- **local-exec**: ejecuta un comando en la máquina donde se ejecuta Terraform.
- **remote-exec**: ejecuta un comando en el recurso remoto vía SSH o WinRM.

### Ejemplo con local-exec

```hcl
    command = "echo ${self.public_ip} >> inventory.txt"
  }

  provisioner "local-exec" {
    command = "ansible-playbook -i '${self.public_ip},' playbook.yml"
  }
}
```


### Provisioners en la destrucción

Se puede configurar un provisioner para que se ejecute cuando el recurso se destruye:

```hcl
resource "aws_instance" "web" {
  # ...

  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Instancia ${self.id} destruida' >> destruction.log"
  }
}
```

### Control de errores

Por defecto, si un provisioner falla, Terraform marca el recurso como **tainted** y lo intentará recrear en el siguiente `apply`. Se puede cambiar este comportamiento:

```hcl
provisioner "local-exec" {
  command    = "some-command-that-might-fail"
  on_failure = continue  # Ignorar el error y continuar
}
```

:::danger[Los provisioners son un ÚLTIMO RECURSO]
HashiCorp recomienda explícitamente **evitar los provisioners** siempre que sea posible. Las razones son:

1. **No son declarativos**: rompen el modelo declarativo de Terraform. No se puede hacer un `plan` de lo que ejecutará un provisioner.
2. **No son idempotentes**: si un script falla a mitad de ejecución y se reintenta, puede dejar el sistema en un estado inconsistente.
3. **Acoplan la configuración**: mezclan la provisión de infraestructura con la configuración del software, que deberían ser responsabilidades separadas.
4. **Dificultan la depuración**: los errores en provisioners son más difíciles de diagnosticar y reproducir.
:::

### Alternativas a los provisioners

Para la gran mayoría de casos, existen alternativas superiores:

| Necesidad | Alternativa recomendada |
|---|---|
| Configurar una instancia EC2 al arrancar | `user_data` con cloud-init |
| Crear una AMI preconfigurada | Packer |
| Configuración post-provision | Ansible, Chef, Puppet |
| Ejecutar un script local tras crear un recurso | `null_resource` + `local-exec` (si no hay otra opción) |
| Notificar a un sistema externo | Webhooks o integraciones nativas del provider |

### Cuándo es aceptable usar provisioners

Los provisioners son aceptables en escenarios muy específicos:

- **Bootstrap inicial** de un cluster donde no hay otra forma de inyectar la configuración.
- **Integración con herramientas legacy** que no tienen provider de Terraform.
- **Pruebas de concepto rápidas** donde la velocidad de desarrollo importa más que la robustez.

Incluso en estos casos, se debería planificar la migración a una solución más robusta.

## 8. Terraform en CI/CD

### Por qué integrar Terraform en pipelines

Ejecutar Terraform manualmente desde la máquina de un desarrollador presenta varios problemas en entornos profesionales:

- **Inconsistencia**: diferentes desarrolladores pueden tener diferentes versiones de Terraform o providers.
- **Falta de auditoría**: no queda registro de quién aplicó qué cambio y cuándo.
- **Sin revisión**: los cambios se aplican sin que otro miembro del equipo los revise.
- **Riesgo de errores**: un `terraform apply` accidental en producción puede ser catastrófico.

La integración en CI/CD resuelve todos estos problemas al proporcionar un entorno controlado, reproducible y auditable.

### Pipeline típico

El flujo estándar de Terraform en un pipeline de CI/CD sigue estos pasos:

1. **fmt** -- Verificar que el código está formateado correctamente.
2. **validate** -- Validar la sintaxis y la configuración.
3. **plan** -- Generar y mostrar el plan de ejecución.
4. **approval** -- (Solo en producción) Aprobación manual por parte del equipo.
5. **apply** -- Aplicar los cambios.

### Ejemplo completo con GitHub Actions

```yaml
name: Terraform

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  TF_VERSION: "1.7.0"
  WORKING_DIR: "./infrastructure"

permissions:
  contents: read
  pull-requests: write
  id-token: write

jobs:
  terraform:
    name: "Terraform Plan & Apply"
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ${{ env.WORKING_DIR }}

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: Configure AWS Credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: eu-west-1

      - name: Terraform Format
        run: terraform fmt -check -recursive

      - name: Terraform Init
        run: terraform init

      - name: Terraform Validate
        run: terraform validate

      - name: Terraform Plan
        id: plan
        run: terraform plan -out=plan.tfplan -no-color
        continue-on-error: true

      - name: Comment Plan on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const output = `#### Terraform Plan
            \`\`\`
            ${{ steps.plan.outputs.stdout }}
            \`\`\`
            `;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });

      - name: Plan Status
        if: steps.plan.outcome == 'failure'
        run: exit 1

      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply plan.tfplan
```

:::tip[Aprobación manual en producción]
En entornos de producción, es fundamental añadir un paso de aprobación manual antes del `apply`. En GitHub Actions, esto se puede conseguir con **environments** que requieran revisión:

```yaml
jobs:
  plan:
    runs-on: ubuntu-latest
    # ... pasos de plan

  apply:
    needs: plan
    runs-on: ubuntu-latest
    environment: production  # Requiere aprobacion manual
    steps:
      # ... pasos de apply
```

Configura el environment `production` en los settings del repositorio para requerir la aprobación de uno o más revisores antes de que el job se ejecute.
:::

### Gestión de secrets en CI/CD

La gestión segura de credenciales es crítica cuando Terraform se ejecuta en un pipeline. Existen varias estrategias:

**OIDC (OpenID Connect) -- Recomendado**

La forma más segura de autenticarse con proveedores cloud desde GitHub Actions es mediante OIDC. No se almacenan credenciales de larga duración:

```yaml
- name: Configure AWS Credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789012:role/github-actions
    aws-region: eu-west-1
```

**Environment secrets**

Para valores que no pueden usar OIDC, GitHub Actions permite definir secrets a nivel de environment:

```yaml
- name: Terraform Init
  env:
    TF_VAR_db_password: ${{ secrets.DB_PASSWORD }}
    TF_VAR_api_key: ${{ secrets.API_KEY }}
  run: terraform init
```

**Nunca almacenar secrets en el state**

Recuerda que el state de Terraform puede contener valores sensibles en texto plano. Es imprescindible:

- Usar un backend remoto con cifrado (S3 + KMS, por ejemplo).
- Restringir el acceso al state con políticas IAM.
- No guardar el state en el repositorio.

## 9. Patrones empresariales

### Workspaces vs directorios: cuándo usar cada uno

La decisión entre workspaces y directorios separados por entorno es una de las más debatidas en la comunidad de Terraform:

**Usar workspaces cuando:**

- Los entornos son prácticamente idénticos y solo difieren en tamaños o cantidades.
- Se trabaja con un equipo pequeño donde la coordinación es sencilla.
- Se quiere un flujo de trabajo simple sin duplicación de código.

**Usar directorios separados cuando:**

- Los entornos tienen diferencias arquitectónicas significativas (por ejemplo, producción tiene multi-AZ y dev no).
- Se necesitan diferentes niveles de acceso por entorno.
- Se quiere poder hacer deploy de un entorno sin riesgo de afectar otros.
- El equipo es grande y múltiples ingenieros trabajan simultáneamente.

### Monorepo vs multi-repo

**Monorepo** (toda la infraestructura en un único repositorio):

- Ventajas: visión global, refactoring más fácil, dependencias entre módulos más claras.
- Desventajas: el repositorio puede crecer mucho, los pipelines de CI/CD son más complejos, los permisos son menos granulares.

**Multi-repo** (un repositorio por componente o equipo):

- Ventajas: ownership claro, pipelines independientes, permisos granulares.
- Desventajas: duplicación de configuración, más difícil mantener la consistencia, dependencias entre repos son más complicadas.

### Estructura de directorio recomendada

Para proyectos de tamaño mediano-grande, la siguiente estructura ofrece un buen equilibrio entre modularidad y simplicidad:

```
infrastructure/
├── modules/                    # Modulos reutilizables
│   ├── vpc/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── eks/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   └── rds/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
├── environments/               # Configuracion por entorno
│   ├── dev/
│   │   ├── main.tf            # Llama a los modulos
│   │   ├── variables.tf
│   │   ├── terraform.tfvars   # Valores especificos de dev
│   │   └── backend.tf         # Backend config para dev
│   ├── staging/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   └── prod/
│       ├── main.tf
│       ├── variables.tf
│       ├── terraform.tfvars
│       └── backend.tf
└── global/                     # Recursos compartidos entre entornos
    ├── iam/
    │   ├── main.tf
    │   ├── variables.tf
    │   └── outputs.tf
    └── dns/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

Los principios clave de esta estructura son:

1. **Módulos en `/modules`**: contienen la lógica reutilizable. No tienen valores hardcodeados; todo se recibe vía variables.
2. **Entornos en `/environments`**: cada entorno tiene su propia configuración que instancia los módulos con los valores apropiados. Cada entorno tiene su propio state.
3. **Global en `/global`**: recursos que son compartidos entre todos los entornos (IAM roles, DNS zones, etc.). Tienen su propio state separado.

Un ejemplo de cómo un entorno instancia los módulos:

```hcl
# environments/dev/main.tf
module "vpc" {
  source = "../../modules/vpc"

  cidr_block         = "10.0.0.0/16"
  availability_zones = ["eu-west-1a", "eu-west-1b"]
  environment        = "dev"
}

module "eks" {
  source = "../../modules/eks"

  vpc_id             = module.vpc.vpc_id
  subnet_ids         = module.vpc.private_subnet_ids
  cluster_version    = "1.28"
  node_instance_type = "t3.medium"
  desired_capacity   = 2
  environment        = "dev"
}

module "rds" {
  source = "../../modules/rds"

  vpc_id            = module.vpc.vpc_id
  subnet_ids        = module.vpc.database_subnet_ids
  instance_class    = "db.t3.micro"
  allocated_storage = 20
  multi_az          = false
  environment       = "dev"
}
```

:::info[No hay una única respuesta correcta]
La estructura ideal depende del contexto de cada equipo: tamaño del equipo, complejidad de la infraestructura, requisitos de seguridad y normativa, y herramientas complementarias que se utilicen. Lo importante es elegir una estructura, documentarla, y mantener la consistencia a lo largo del tiempo. Es mejor una estructura simple que todo el equipo entiende que una estructura "perfecta" que nadie sigue.
:::

### Checklist para proyectos empresariales

Independientemente de la estructura elegida, un proyecto de Terraform empresarial debería cumplir con estos requisitos:

- **Backend remoto** con cifrado y locking (S3 + DynamoDB, Terraform Cloud, etc.).
- **State separado** por entorno y por componente de infraestructura.
- **Módulos versionados** para garantizar la reproducibilidad.
- **Pipeline de CI/CD** que ejecute plan en PRs y apply solo tras aprobación.
- **Política de branching** clara (por ejemplo, trunk-based development).
- **Code review obligatorio** para cualquier cambio en infraestructura.
- **Documentación** de decisiones arquitectónicas y convenciones del equipo.

## 10. Resumen

Los conceptos cubiertos en esta unidad son fundamentales para trabajar con Terraform a escala profesional:

- **Workspaces** permiten gestionar múltiples entornos con la misma configuración, cada uno con su propio state aislado. Son útiles para casos simples, pero los directorios separados ofrecen mayor control en proyectos complejos.

- **for_each** es la forma preferida de crear múltiples instancias de un recurso cuando los elementos tienen identidad propia. Proporciona estabilidad ante cambios en la colección, al contrario que `count`.

- **count** es útil para crear N copias idénticas de un recurso y, especialmente, para el patrón condicional `count = condition ? 1 : 0`.

- **Dynamic blocks** permiten generar bloques repetidos dentro de un recurso de forma programática, aunque no se debe abusar de ellos por razones de legibilidad.

- **Expresiones condicionales** con el operador ternario permiten adaptar la configuración según variables o el entorno.

- **Funciones built-in** ofrecen un amplio repertorio para manipular strings, colecciones, números, ficheros y datos de red. El comando `terraform console` es invaluable para experimentar con ellas.

- **Provisioners** ejecutan scripts en los recursos, pero deben evitarse en favor de alternativas como user_data, Packer o Ansible.

- **CI/CD** proporciona consistencia, auditoría y control de cambios. Un pipeline típico sigue el flujo fmt, validate, plan, approval, apply.

- **Patrones empresariales** como la estructura de directorios con módulos compartidos, el uso de backends remotos y las políticas de code review son esenciales para gestionar infraestructura a escala.

Estos patrones y herramientas, combinados con los fundamentos de las unidades anteriores, proporcionan una base sólida para gestionar infraestructura como código en entornos reales de producción.
