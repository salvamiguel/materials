# Módulos y Reutilización

Hasta ahora hemos trabajado con configuraciones Terraform contenidas en un único directorio. Esto funciona para proyectos pequeños, pero a medida que la infraestructura crece, mantener todo en un solo lugar se vuelve inmanejable. Los módulos son la herramienta fundamental de Terraform para organizar, reutilizar y estandarizar configuraciones de infraestructura.

---

## 1. Qué es un módulo

Un **módulo** en Terraform es, en su forma más simple, un directorio que contiene archivos `.tf`. Cualquier conjunto de archivos Terraform en un directorio constituye un módulo. Cuando ejecutas `terraform plan` o `terraform apply`, estás ejecutando un módulo: el **root module**.

### Root module vs. child modules

- **Root module**: es el directorio de trabajo donde ejecutas los comandos de Terraform. Contiene tu configuración principal y es el punto de entrada de toda la ejecución.
- **Child modules**: son módulos que el root module (u otros módulos) invocan mediante el bloque `module`. Pueden estar en subdirectorios locales, en un registry o en un repositorio Git remoto.

### La analogía con funciones

Piensa en los módulos como **funciones en programación**:

| Concepto en programación | Equivalente en Terraform |
|---|---|
| Parámetros de entrada | `variable` (input variables) |
| Cuerpo de la función | Recursos y data sources en `main.tf` |
| Valor de retorno | `output` (output values) |
| Llamada a función | Bloque `module` |

Al igual que una buena función, un buen módulo tiene una interfaz clara, una responsabilidad definida y puede invocarse múltiples veces con distintos argumentos.

### Ventajas de usar módulos

- **Reutilización**: escribe la lógica una vez, úsala en múltiples entornos y proyectos.
- **Encapsulación**: oculta la complejidad interna. El consumidor del módulo solo necesita conocer las variables de entrada y los outputs.
- **Estandarización**: establece patrones aprobados por el equipo. Si todos usan el mismo módulo de red, todas las redes siguen las mismas convenciones.
- **Testing**: un módulo aislado es más fácil de probar que una configuración monolítica.
- **Colaboración**: distintos equipos pueden mantener distintos módulos de forma independiente.

## 2. Estructura de un módulo

Terraform no impone una estructura de archivos dentro de un módulo, pero la comunidad ha establecido una convención clara que deberías seguir:

```
modules/webapp/
├── main.tf          # Recursos principales
├── variables.tf     # Input variables
├── outputs.tf       # Output values
├── versions.tf      # Required providers y versiones de Terraform
└── README.md        # Documentación del módulo
```

### Responsabilidad de cada archivo

- **`main.tf`**: contiene los recursos y data sources que el módulo gestiona. En módulos más grandes, puedes dividir los recursos en varios archivos (`networking.tf`, `compute.tf`, etc.), pero `main.tf` siempre debe existir como punto de entrada principal.
- **`variables.tf`**: declara todas las input variables del módulo. Cada variable debe tener `description` y `type` definidos.
- **`outputs.tf`**: expone valores que otros módulos o el root module pueden consumir. Piensa en los outputs como la API pública de tu módulo.
- **`versions.tf`**: especifica la versión mínima de Terraform y los providers requeridos.
- **`README.md`**: documenta el propósito del módulo, sus variables, outputs y ejemplos de uso.

:::tip
Mantén cada módulo **pequeño y enfocado** en una sola responsabilidad. Un módulo que crea una red no debería crear también instancias de cómputo. Aplica el principio de single responsibility: si necesitas describir tu módulo con "y" (por ejemplo, "crea una red **y** despliega una aplicación"), probablemente deberías dividirlo en dos módulos.
:::

resource "aws_security_group" "webapp" {

  description = "Security group para ${var.app_name}"

      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]

  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]

    Name = "${var.app_name}-sg"

}

resource "aws_instance" "webapp" {

### Meta-argumentos en modules

Los bloques `module` soportan varios meta-argumentos que amplían su funcionalidad:

**`count`** -- crear múltiples instancias del módulo:

```hcl
module "webapp" {
  source = "./modules/webapp"
  count  = 3

  app_name = "app-${count.index}"
  # ... demás argumentos según el provider
}

# Acceso: module.webapp[0].public_ip
```

**`for_each`** -- crear instancias basadas en un map o set:

```hcl
variable "apps" {
  default = {
    frontend = { ports = [80, 443] }
    backend  = { ports = [8080] }
    admin    = { ports = [443, 8443] }
  }
}

module "webapp" {
  source   = "./modules/webapp"
  for_each = var.apps

  app_name      = each.key
  allowed_ports = each.value.ports
  # ... demás argumentos según el provider
}

# Acceso: module.webapp["frontend"].public_ip
```

**`depends_on`** -- establecer dependencias explícitas:

```hcl
module "webapp" {
  source = "./modules/webapp"
  # ...

  depends_on = [module.network]
}
```

**`providers`** -- pasar configuraciones de provider específicas:

<CloudTabs>
<CloudTab provider="aws">

```hcl
module "webapp_eu" {
  source = "./modules/webapp"
  # ...

  providers = {
    aws = aws.europe
  }
}
```

</CloudTab>
<CloudTab provider="gcp">

```hcl
module "webapp_eu" {
  source = "./modules/webapp"
  # ...

  providers = {
    google = google.europe
  }
}
```

</CloudTab>
<CloudTab provider="azure">

```hcl
module "webapp_eu" {
  source = "./modules/webapp"
  # ...

  providers = {
    azurerm = azurerm.europe
  }
}
```

</CloudTab>
</CloudTabs>

:::info
Usa `for_each` en lugar de `count` siempre que sea posible. Con `count`, si eliminas un elemento del medio de la lista, Terraform recreará los elementos posteriores. Con `for_each` y claves estables, cada instancia se identifica por su clave, no por su posición.
:::

## 5. Fuentes de módulos

Terraform soporta múltiples fuentes para cargar módulos. La elección depende de tu flujo de trabajo y del grado de compartición que necesites.

### Módulos locales

La forma más sencilla. El módulo vive en un subdirectorio del mismo repositorio:

```hcl
module "vpc" {
  source = "./modules/vpc"
  # ...
}
```

Ventajas: simplicidad, sin dependencias externas, cambios atómicos con el resto del código.

### Terraform Registry

El [Terraform Registry](https://registry.terraform.io/) es el repositorio público de módulos de la comunidad. Contiene módulos verificados y mantenidos activamente.

Estos módulos oficiales crean una red completa con subnets, routing y todo lo necesario, en pocas líneas. Sin módulos, esta configuración requeriría cientos de líneas de HCL.

Para explorar módulos disponibles:

```bash
# Buscar módulos en el registry desde la CLI
terraform providers
# O visita https://registry.terraform.io/browse/modules
```

### GitHub

Puedes referenciar módulos directamente desde repositorios de GitHub:

```hcl
module "vpc" {
  source = "github.com/mi-org/terraform-modules//modules/vpc?ref=v2.1.0"
}
```

La doble barra `//` separa el repositorio de la ruta al subdirectorio dentro del repositorio. El parámetro `ref` especifica el tag, branch o commit.

### Git genérico

Para repositorios Git que no están en GitHub, usa el prefijo `git::`:

```hcl
module "vpc" {
  source = "git::https://gitlab.com/mi-org/terraform-modules.git//vpc?ref=v1.0.0"
}

# Con SSH
module "vpc" {
  source = "git::ssh://git@gitlab.com/mi-org/terraform-modules.git//vpc?ref=v1.0.0"
}
```

### Otros sources

Terraform también soporta:
- **S3 bucket**: `source = "s3::https://s3-eu-west-1.amazonaws.com/bucket/vpc.zip"`
- **GCS bucket**: `source = "gcs::https://www.googleapis.com/storage/v1/bucket/vpc.zip"`
- **HTTP URL**: `source = "https://example.com/modules/vpc.zip"`

:::info
**Siempre** especifica la versión al usar módulos remotos. Sin versión fija, Terraform descargará la última versión disponible, lo que puede introducir breaking changes inesperados. Usa `version` para el Registry y `ref` para fuentes Git.
:::

Después de añadir o cambiar un `source`, es necesario ejecutar:

```bash
terraform init
```

Este comando descarga los módulos referenciados y los almacena en el directorio `.terraform/modules/`.

## 6. Inputs con validación y tipos complejos

Las variables de un módulo pueden ir mucho más allá de tipos primitivos. Terraform permite definir estructuras complejas con validaciones que atrapan errores antes de que lleguen a la API del provider.

### Variables con tipo object

```hcl
variable "database" {
  type = object({
    engine            = string
    engine_version    = string
    instance_class    = string
    allocated_storage = number
  })

  description = "Configuración de la base de datos RDS."

  validation {
    condition     = contains(["mysql", "postgres"], var.database.engine)
    error_message = "Engine debe ser mysql o postgres."
  }

  validation {
    condition     = var.database.allocated_storage >= 20
    error_message = "El almacenamiento mínimo es 20 GB."
  }
}
```

Este patrón agrupa configuraciones relacionadas en un solo objeto, haciendo la interfaz del módulo más limpia y evitando una explosión de variables individuales.

Uso desde el consumidor:

```hcl
module "rds" {
  source = "./modules/rds"

  database = {
    engine            = "postgres"
    engine_version    = "15.4"
    instance_class    = "db.t3.micro"
    allocated_storage = 20
  }
}
```

### Variables con tipo map de objects

Para escenarios donde necesitas crear múltiples recursos con configuraciones distintas:

```hcl
variable "security_group_rules" {
  type = map(object({
    port        = number
    protocol    = string
    cidr_blocks = list(string)
  }))

  default = {
    http = {
      port        = 80
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
    https = {
      port        = 443
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}
```

### Variables con optional()

A partir de Terraform 1.3, puedes marcar atributos de un object como opcionales y asignarles valores por defecto:

```hcl
variable "app_config" {
  type = object({
    name        = string
    replicas    = optional(number, 1)
    environment = optional(string, "production")
    labels      = optional(map(string), {})
  })

  description = "Configuración de la aplicación con valores opcionales."
}
```

El consumidor solo necesita proporcionar `name`; el resto toma valores por defecto:

```hcl
module "app" {
  source = "./modules/app"

  app_config = {
    name = "mi-servicio"
    # replicas, environment y labels usan sus defaults
  }
}
```

### Validaciones avanzadas

```hcl
variable "environment" {
  type        = string
  description = "Entorno de despliegue."

  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "Environment debe ser dev, staging o production."
  }
}

variable "cidr_block" {
  type        = string
  description = "CIDR block para la red."

  validation {
    condition     = can(cidrhost(var.cidr_block, 0))
    error_message = "El valor debe ser un CIDR block válido (ej: 10.0.0.0/16)."
  }
}
```

:::tip
Las validaciones son tu primera línea de defensa. Es mucho mejor que Terraform falle durante `plan` con un mensaje claro que esperar un error críptico del API del provider durante `apply`.
:::

## 7. Composición de módulos

El verdadero poder de los módulos aparece cuando los **compones**: el root module orquesta varios child modules, pasando outputs de uno como inputs de otro.

### Patrón de composición

Imagina un proyecto con tres componentes: red, aplicación y base de datos. Cada uno es un módulo independiente, y el root module los conecta.

```
proyecto/
├── main.tf              # Root module: orquesta los child modules
├── variables.tf         # Variables globales del proyecto
├── outputs.tf           # Outputs del proyecto
├── terraform.tfvars     # Valores concretos
└── modules/
    ├── network/
    │   ├── main.tf
    │   ├── variables.tf
    │   └── outputs.tf
    ├── webapp/
    │   ├── main.tf
    │   ├── variables.tf
    │   └── outputs.tf
    └── database/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

### Root module (main.tf)

### Múltiples entornos con la misma composición

Una vez que tienes la composición, puedes reutilizarla para distintos entornos usando diferentes `.tfvars`:

```bash
# Desplegar en desarrollo
terraform apply -var-file="environments/dev.tfvars"

# Desplegar en producción
terraform apply -var-file="environments/prod.tfvars"
```

Donde `dev.tfvars` y `prod.tfvars` contienen valores distintos para las mismas variables:

```hcl
# environments/dev.tfvars
environment   = "dev"
instance_type = "t3.small"
vpc_cidr      = "10.0.0.0/16"

# environments/prod.tfvars
environment   = "production"
instance_type = "t3.xlarge"
vpc_cidr      = "10.1.0.0/16"
```

## 8. Versionado de módulos

Cuando un módulo se comparte entre equipos o proyectos, el versionado es esencial. Sin él, un cambio en el módulo puede romper todas las configuraciones que lo consumen.

### Git tags

La forma más directa de versionar un módulo en un repositorio Git es mediante tags:

```bash
# Crear una versión
git tag v1.0.0
git push origin v1.0.0

# Crear una versión con mensaje
git tag -a v1.1.0 -m "Añade soporte para múltiples subnets"
git push origin v1.1.0
```

El consumidor referencia la versión exacta:

```hcl
module "vpc" {
  source = "git::https://github.com/mi-org/terraform-vpc.git?ref=v1.0.0"
}
```

### Terraform Registry y version constraints

Cuando el módulo está publicado en el Terraform Registry, usa el atributo `version`:

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.5.1"
  # ...
}
```

Los constraints de versión siguen la misma sintaxis que los providers:

| Constraint | Significado |
|---|---|
| `= 5.5.1` | Exactamente la versión 5.5.1 |
| `>= 5.0.0` | Versión 5.0.0 o superior |
| `~> 5.5.0` | Versión >= 5.5.0 y < 5.6.0 (patch updates) |
| `~> 5.0` | Versión >= 5.0.0 y < 6.0.0 (minor updates) |
| `>= 5.0.0, < 6.0.0` | Rango explícito |

### Semantic versioning (SemVer)

Los módulos deberían seguir [Semantic Versioning](https://semver.org/):

- **MAJOR** (v2.0.0): cambios que rompen compatibilidad (eliminar una variable, cambiar un tipo de output).
- **MINOR** (v1.1.0): nueva funcionalidad compatible hacia atrás (añadir una variable con default).
- **PATCH** (v1.0.1): corrección de bugs sin cambiar la interfaz.

:::warning
**Nunca** apuntes a `main` o `master` en producción. Un cambio que un compañero pushee al branch principal puede romper tu infraestructura en el próximo `terraform apply`. Siempre usa tags o versiones fijas. Incluso en desarrollo, es recomendable usar una referencia concreta para evitar sorpresas.
:::

### Estrategia de actualización

Cuando necesites actualizar un módulo:

```bash
# 1. Cambia la versión en tu configuración
# version = "5.5.1" -> version = "5.6.0"

# 2. Descarga la nueva versión
terraform init -upgrade

# 3. Revisa los cambios que se aplicarían
terraform plan

# 4. Si todo es correcto, aplica
terraform apply
```

## 9. Buenas prácticas

Estas convenciones han sido establecidas por la comunidad de Terraform y facilitan la adopción, mantenimiento y colaboración en módulos compartidos.

### Naming

Sigue la convención de nomenclatura del Terraform Registry:

```
terraform-<PROVIDER>-<NAME>
```

Ejemplos:
- `terraform-aws-vpc`
- `terraform-google-kubernetes-engine`
- `terraform-azurerm-storage-account`

Esta convención permite que el Registry detecte automáticamente el provider principal del módulo.

### Documentación

Cada módulo debe incluir un `README.md` con:
- Descripción del propósito del módulo.
- Requisitos previos (versión de Terraform, providers).
- Tabla de variables con descripciones y valores por defecto.
- Tabla de outputs.
- Al menos un ejemplo de uso completo.

Herramientas como [terraform-docs](https://terraform-docs.io/) generan esta documentación automáticamente a partir de los archivos `.tf`:

```bash
# Instalar terraform-docs
brew install terraform-docs

# Generar documentación
terraform-docs markdown table ./modules/webapp > ./modules/webapp/README.md
```

### Parametrización

No hardcodees valores dentro de un módulo. Todo lo que pueda variar entre usos debe ser una variable:

```hcl
# Mal: valor hardcodeado
resource "aws_instance" "web" {
  instance_type = "t3.large"
  # ...
}

# Bien: parametrizado
resource "aws_instance" "web" {
  instance_type = var.instance_type
  # ...
}
```

Proporciona `default` solo cuando existe un valor sensato para la mayoría de los casos de uso.

### Outputs completos

Expón como output todo lo que otro módulo pueda necesitar. Es mejor tener outputs de más que de menos. Si alguien que consume tu módulo tiene que hacer un `terraform state show` para obtener un atributo, probablemente falta un output:

```hcl
# Exponemos todo lo relevante del recurso
output "instance_id" {
  value       = aws_instance.web.id
  description = "ID de la instancia."
}

output "instance_arn" {
  value       = aws_instance.web.arn
  description = "ARN de la instancia."
}

output "public_ip" {
  value       = aws_instance.web.public_ip
  description = "IP pública."
}

output "private_ip" {
  value       = aws_instance.web.private_ip
  description = "IP privada."
}
```

### Mantener módulos genéricos

Un módulo debe ser reutilizable en distintos contextos. Evita asumir detalles del entorno:

- No asumas nombres de redes, regiones o cuentas del provider.
- No incluyas configuración de backend (el root module se encarga).
- No declares bloques `provider` dentro del módulo (usa `required_providers` en su lugar).

### Checklist de calidad

Antes de publicar o compartir un módulo, verifica:

- [ ] Todas las variables tienen `type` y `description`.
- [ ] Todos los outputs tienen `description`.
- [ ] Existe `versions.tf` con las versiones mínimas requeridas.
- [ ] No hay valores hardcodeados que deberían ser variables.
- [ ] El módulo funciona de forma aislada (no depende de recursos externos no declarados).
- [ ] Existe un `README.md` con al menos un ejemplo de uso.
- [ ] Sigue la convención de naming `terraform-<PROVIDER>-<NAME>`.
- [ ] Los cambios respetan semantic versioning.

## 10. Resumen

Los módulos son el mecanismo fundamental de Terraform para construir infraestructura mantenible a escala. Estos son los puntos clave de esta unidad:

- Un **módulo** es un directorio con archivos `.tf`. El directorio donde ejecutas Terraform es el **root module**; los que invocas con el bloque `module` son **child modules**.
- La estructura estándar incluye `main.tf`, `variables.tf`, `outputs.tf` y `versions.tf`.
- Los módulos se **consumen** con el bloque `module`, pasando valores a las variables de entrada y accediendo a los resultados mediante `module.<nombre>.<output>`.
- Los meta-argumentos `count`, `for_each`, `depends_on` y `providers` permiten crear múltiples instancias de un módulo y controlar dependencias.
- Las **fuentes** pueden ser locales, del Terraform Registry, de GitHub o de cualquier repositorio Git. Siempre fija la versión en entornos de producción.
- Las **validaciones** en variables atrapan errores antes de que lleguen al provider, mejorando la experiencia del consumidor del módulo.
- La **composición** de módulos permite orquestar infraestructura compleja conectando outputs de un módulo como inputs de otro.
- El **versionado** con SemVer y Git tags protege a los consumidores de breaking changes inesperados.
- Las **buenas prácticas** (naming, documentación, parametrización, outputs completos) son tan importantes como el código en sí.

Con una buena estrategia de módulos, tu infraestructura Terraform se convierte en una librería de componentes reutilizables que todo el equipo puede consumir con confianza.
